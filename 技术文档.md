# 中驰智能PPT生成平台技术文档

## 1. 项目背景

本项目目标是建设一套面向客户、业务部门、售前人员使用的智能PPT生成平台。用户不需要理解复杂的PPT制作逻辑，只需要在网页中创建项目、填写基础信息、统一上传项目资料，系统即可完成资料解析、项目类型识别、模板选择、案例匹配、字段替换、PPT合并，并最终输出可交付的PPT初稿。

本阶段仍以 Demo 闭环为主，但技术口径必须贴近后续生产系统，避免继续沿用“前端按模块上传”的旧方案。

## 2. 当前最新业务口径

本节是 2026-05-13 后续修正后的最新口径，优先级高于旧文档和旧 Demo 中的任何描述。

前端不再按 `M1/M2/M5/M6` 分模块上传资料。前端只负责：

- 创建项目。
- 填写项目基础信息。
- 一次性统一上传全部项目资料。
- 展示系统识别结果。
- 让用户人工确认或修正识别结果。
- 展示生成状态。
- 下载最终 PPTX。

后端负责：

- 保存全部上传资料。
- 解析资料文本和元信息。
- 自动判断资料角色和可服务模块。
- 识别项目类型。
- 为 `M1/M2` 选择对应固化模板。
- 为 `M5` 匹配同类型案例。
- 为 `M6` 使用固定企业背书模板。
- 生成并合并最终 PPTX。

模块处理方式如下：

| 模块 | 最新处理方式 | 说明 |
|---|---|---|
| M1 行业背景与技术标准 | 固化模板字段替换 | 与 M2 共用项目类型模板，不从零动态生成 |
| M2 行业痛点与问题分析 | 固化模板字段替换 | 先识别公路、铁路、地铁、轨交既有线等类型，再替换字段 |
| M3 定制化设计方案（工程概况与需求分析） | 固化模板文字替换，图片替换独立功能测试 | 正式流程继续使用现有 M3 文字替换模板；图片替换只在功能测试中验证，不影响正式流程 |
| M4 （核心解决方案与技术优化） | 后续动态生成 | 当前 Demo 不生成，后续根据工程量、用钢量、工期规则生成 |
| M5 同类型案例匹配 | 根据资料找案例 | 从资料中提取标签，匹配历史案例并填入案例模板 |
| M6 企业介绍与荣誉背书 | 固定模板 | 默认使用企业介绍/资质/荣誉模板，可补充替换字段 |

## 3. 新版主流程

1. 用户创建项目，填写项目名称、所在地、业主单位、产品线等基础信息。
2. 用户统一上传全部项目资料，不选择资料属于 M 几。
3. 后端保存原始文件，生成文件记录。
4. 后端解析文件内容，抽取文本、文件名、页标题、表格信息和可用元数据。
5. 后端判断每个文件的资料角色，例如项目简介、图纸、招标文件、调研表、案例材料、企业材料、未知材料。
6. 后端识别项目类型：公路、铁路、地铁、轨交既有线等。
7. 后端为 `M1/M2` 选择对应固化模板。
8. 后端为 `M5` 提取项目标签并匹配同类型案例。
9. 后端为 `M6` 选择固定企业背书模板。
10. 前端展示识别结果，用户确认或修正项目类型、模板选择、案例选择和缺失字段。
11. 后端按确认结果执行模板字段替换、M3 文字替换、案例填充和固定模板复制。
12. 后端按 `M1/M2 -> M3 -> M5 -> M6` 合并最终 PPTX；用户选择跳过 M3 或不选择 M5 时，对应章节不参与合并。
13. 用户下载最终 PPTX。

旧 Demo 中的 `/api/projects/{project_id}/modules/{module_id}/files` 可暂时保留为兼容接口，但不再作为新版主流程入口，前端新流程不得调用该接口。

## 4. 技术选型

| 模块 | 技术选型 | 说明 |
|---|---|---|
| 前端 | Next.js | 项目创建、统一上传、识别结果确认、生成状态、下载 |
| 后端 | FastAPI + uv | API 服务、文件记录、任务状态、生成调度 |
| 工作流 | LangGraph 或可替换工作流 | 编排资料解析、结构化分类、类型识别、模板选择、案例匹配、人工确认、PPT生成 |
| 数据库 | PostgreSQL | 生产存储项目、文件、解析结果、任务、模板、案例、生成结果 |
| 向量库 | pgvector | 分阶段用于 M5 案例召回、素材召回和项目资料 chunk 检索，不用于让前端分模块 |
| 文件存储 | 本地目录起步，生产换 MinIO/S3 | Demo 先降低复杂度 |
| PPT生成 | python-pptx + 固化模板 | 复制模板、字段替换、案例填充、PPT合并 |
| 文档解析 | Python 解析库 + 后续 OCR/CAD 解析 | PDF、Word、Excel、PPT 先做基础文本与结构抽取；图片、扫描件、CAD 标记为待增强 |
| LLM服务 | OpenAI兼容API或企业内网模型，支持 Mock | 用于辅助摘要、标签提取、类型判断、缺失字段识别、案例匹配理由，不允许编造关键工程数据 |
| Embedding服务 | OpenAI兼容Embedding或企业内网Embedding模型 | 先用于案例匹配，后续扩展到素材和文档 chunk 检索；Demo 可先关键词 fallback |

Python 项目默认使用 `uv` 管理环境和依赖。涉及深度学习环境时，才使用本机已有 miniconda 环境。

## 5. 模板策略

固化模板统一配置：

```text
PPT_TEMPLATE_ROOT=D:\中驰股份\code\ppt_engine\templates\solution_fixed_modules
```

`M1/M2` 候选模板：

| 项目类型 | project_type | 模板文件 |
|---|---|---|
| 公路 | `highway` | `公路全封闭声屏障（M1_&_M2）.pptx` |
| 地铁 | `metro` | `轨道交通地铁全封闭声屏障（M1_&_M2）.pptx` |
| 铁路/轨交既有线 | `existing_rail_transit` | `铁路_&_轨道交通既有线声屏障_（M1_&_M2）.pptx` |
| 铁路 | `railway` | `铁路声屏障行业背景与技术发展（M1_&_M2）.pptx` |

`M5` 案例模板：

```text
%PPT_TEMPLATE_ROOT%\南昌轨道交通4号线声屏障工程项目案例模板（M5）.pptx
```

`M6` 固定模板：

```text
%PPT_TEMPLATE_ROOT%\中驰企业介绍合并初版（M6）.pptx
```

`M3` 正式文字替换模板：

```text
%PPT_TEMPLATE_ROOT%\M3_项目深化方案模板.pptx
```

`M3` 图片替换独立测试模板：

```text
%PPT_TEMPLATE_ROOT%\M3_项目深化方案模板_图片替换测试.pptx
```

模板实现原则：

- `M1/M2` 不用 AI 从零生成整章。
- `M1/M2` 只做项目类型识别、模板选择、字段替换。
- `M5` 根据项目资料匹配案例，不编造案例。
- `M6` 默认固定使用企业介绍模板，有补充资料时才替换字段或素材。
- `M3` 正式流程继续使用 `M3_项目深化方案模板.pptx` 做文字占位符替换。
- `M3` 图片替换只使用 `M3_项目深化方案模板_图片替换测试.pptx`，不得修改正式 M3 模板。
- M3 图片测试模板中的图片槽必须是“替换槽位”，不是叠在旧图上的遮罩；应清掉旧图片或旧图形，只保留带 `{{image:...}}` 的矩形/图片占位槽。
- 图片占位槽的位置和尺寸就是最终插图的位置和尺寸。PowerPoint 中若用矩形或文本框承载占位符，应关闭“根据文字调整形状大小/Resize shape to fit text”，避免高度被文字锁住。
- 缺失字段统一写成 `[待补充：字段名]`。
- `M4` 本阶段不输出章节。

M3 图片替换测试用途固定为：

| 中文用途 | 后端字段名 | 说明 |
|---|---|---|
| 项目建设范围图 | `project_scope_map` | 单图替换 |
| 项目线路图 | `project_line_map` | 单图替换 |
| 踏勘路线/点位图 | `survey_route_map` | 单图替换 |
| 现场踏勘照片组 | `site_survey_photos` | 多图，按槽位顺序替换，第一版最多 4 张 |
| 重难点证据图 | `key_difficulty_evidence` | 单图替换 |

文件上传时不要求前端提供 `module_id`。

```json
{
  "file_id": 1,
  "project_id": 1,
  "filename": "项目资料.pdf",
  "content_type": "application/pdf",

  "parsed_text_path": "",
  "confidence": 0.76,
  "evidence": [
    {
      "source": "filename|page|sheet|slide|paragraph",
      "keyword": "轨道交通",
      "snippet": "命中片段"
    }
  ],
  "text_chunks_count": 12,
  "missing_field_hints": ["线路名称", "现场痛点"],
  "error_message": ""
}
```

多格式解析要求：

- PDF：优先提取可复制文本；扫描件或图片型 PDF 标记为 `pending_ocr`，后续接 OCR。
- Word：提取正文、标题、表格文字和段落结构。
- Excel：提取 sheet 名、表头、关键单元格、工程量表和参数表。
- PPT：提取每页标题、正文、备注和图片占位信息，供历史案例和素材索引使用。
- 图片：先记录文件名、尺寸和可能用途，后续接 OCR 或图像理解。
- CAD：先保存原文件并标记为 `pending_cad_parse`，后续接 CAD 解析或人工标注。

## 8. 工作流节点

推荐工作流节点：

1. `load_project`：加载项目基础信息。
2. `store_uploaded_files`：保存统一上传的项目资料。
3. `parse_documents`：解析 PDF、Word、Excel、PPT 文本和结构；图片、扫描件、CAD 先标记待增强。
4. `chunk_documents`：按文件、页码、sheet、slide、段落生成可追溯文本 chunk。
5. `classify_documents`：判断资料角色、`assigned_modules`、置信度和命中依据。
6. `detect_project_type`：识别项目类型并输出置信度、命中关键词和证据片段。
7. `select_m1_m2_template`：按项目类型选择固化模板。
8. `extract_case_tags`：提取 M5 案例匹配标签，包括城市、线路、施工场景、痛点、技术难点。
9. `match_m5_cases`：从固定案例库和历史资料中召回真实案例；优先关键词/标签 fallback，后续接 pgvector。
10. `select_m6_template`：选择固定企业模板。
11. `detect_missing_fields`：识别生成前需要人工确认或补充的字段。
12. `wait_human_review`：等待人工确认或修正。
13. `render_m1_m2_from_template`：复制模板并替换字段。
14. `render_m3_from_template`：复制 M3 正式文字模板并替换 active 文本占位符。
15. `render_m5_case_template`：填充确认案例。
16. `render_m6_fixed_template`：复制并补充企业背书模板。
17. `merge_final_ppt`：按 `M1/M2 -> M3 -> M5 -> M6` 合并，跳过未选择章节。
18. `quality_check`：检查最终文件存在、可打开、章节顺序正确、缺失字段有占位、案例来源可追溯。

## 9. LLM 与向量库边界

### 9.1 LLM 可用位置

- 资料摘要：从解析文本中提炼摘要、关键字段和疑似缺失项。
- 资料角色判断：辅助判断 `project_brief`、`drawing`、`tender`、`survey`、`case_material`、`enterprise_material`、`unknown`。
- 标签提取：提取城市、线路、交通类型、施工场景、痛点、技术难点、工期条件。
- 项目类型判断：辅助判断 `highway`、`railway`、`metro`、`existing_rail_transit`，但最终必须落在固定枚举内。
- 案例匹配理由：基于真实案例库、历史资料和命中证据生成解释。
- 缺失字段识别：指出生成 PPT 前需要人工确认的字段。

### 9.2 LLM 禁用或限制位置

- 不允许编造案例、业绩、资质、荣誉、工程量、用钢量、工期。
- 不允许让 LLM 从零生成 `M1/M2` 整章内容；`M1/M2` 只选固化模板并做字段替换。
- 不允许让 LLM 从零生成 `M6` 企业背书内容；`M6` 默认使用固定模板。
- 不允许 LLM 替代人工确认；识别结果必须可由用户确认或修正。
- LLM 不可用时，Demo 必须通过关键词、规则和 Mock fallback 跑通。

### 9.3 向量库分阶段

1. 阶段一：关键词/标签 fallback，保证没有 Embedding 和 pgvector 时仍可演示。
2. 阶段二：对案例摘要和案例 chunk 建 embedding，优先服务 `M5` 案例召回。
3. 阶段三：把素材、历史 PPT、项目资料 chunk 纳入统一检索，用于资料复用、案例推荐和后续 `M3/M4` 动态模块。

## 11. 验收标准

新版 Demo 完成后必须证明：

- 前端不再要求用户选择 M 几上传。
- 用户可以统一上传多个项目资料文件。
- 后端文件记录不强制依赖前端传入 `module_id`。
- 系统能输出资料分类结果。
- 系统能识别项目类型并给出命中关键词和置信度。
- 系统能为 `M1/M2` 选择固化模板。
- 系统能为 `M5` 推荐案例。
- 系统能为 `M6` 选择固定模板。
- 用户能确认或修正识别结果。
- 最终 PPTX 按 `M1/M2 -> M3 -> M5 -> M6` 合并，跳过未选择章节。
- `M3` 文字替换测试和正式 M3 流程不受图片替换测试影响。
- `M3` 图片替换测试只影响功能测试页和 `m3_image_test` 输出目录。
- `M4` 不参与当前生成链路。
- 没有真实 LLM、Embedding、pgvector 时，Demo 可用关键词和 Mock fallback 跑通。
- 资料解析结果包含角色、可服务模块、置信度、命中依据和缺失字段线索。
- 案例推荐必须来自固定案例库或历史资料目录，且包含来源路径或可追溯证据。
- 向量库优先用于 M5 案例召回，再扩展到素材和文档 chunk。
- 大模型接入必须可关闭，不可替代人工确认。

## 12. 测试建议

有关测试只跑相关测试，不跑全量测试。

- 后端：优先运行 `uv run python -m unittest -v tests.test_api`，覆盖统一上传、文件解析、分类、识别、确认、生成、下载。
- M3 图片替换：优先运行 `uv run python -m unittest tests.test_m3_image_renderer tests.test_m3_renderer -v`，同时覆盖图片测试和现有 M3 文字替换不受影响。
- 后端 M3 图片接口：优先运行 `uv run python -m unittest tests.test_m3_image_api -v`。
- PPT 引擎：优先运行 `uv run python -m unittest -v tests.test_renderer`，覆盖模板选择、M5 有/无案例、M6 固定模板、最终 PPTX 可打开。
- 工作流：优先运行 `uv run python -m unittest -v tests.test_workflow`，覆盖资料解析、案例匹配、人工确认、M3/M4 排除。
- 前端：优先运行 `npm test` 和 `.\node_modules\.bin\tsc.cmd --noEmit`，覆盖统一上传、识别结果展示、人工确认、生成与下载入口。
- 当前 Windows/Codex 桌面环境中，除非明确需要生产构建，不要把 `npm run build` 作为快速验证命令。
