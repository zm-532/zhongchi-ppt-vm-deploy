# 中驰智能PPT Demo 文档修订与后续代码改造计划

## 1. 目标

本计划用于交给其他 AI 或工程 Agent 开始后续代码改造，也可供 Codex 后续做代码审查。Agent 只读取 `D:\中驰股份\计划.md` 和 `D:\中驰股份\技术文档.md`，应能理解最新业务口径、接口方向、模块边界、验收标准和分块修改提示词。

本次 Codex 执行范围只修改文档，不修改代码。

## 2. 最新业务理解

前端不再负责判断资料属于 `M1/M2/M5/M6`。用户只需要创建项目、填写基础信息、统一上传全部资料。资料上传后，由后端自动完成解析、资料归类、项目类型识别、模板选择和案例匹配。

模块规则：

| 模块 | 当前处理方式 | 说明 |
|---|---|---|
| M1 行业背景与技术标准 | 固化模板字段替换 | 与 M2 共用项目类型模板 |
| M2 项目概况与现场挑战 | 固化模板字段替换 | 根据公路、铁路、地铁、轨交既有线等类型选择模板 |
| M3 定制化设计方案 | 后续动态生成 | 本阶段不开发、不生成 |
| M4 工程量与施工周期测算 | 后续动态生成 | 本阶段不开发、不生成 |
| M5 同类型案例匹配 | 根据资料匹配案例 | 提取项目标签后找历史案例 |
| M6 企业背书与荣誉 | 固定模板 | 默认使用企业介绍、资质、荣誉模板 |

核心原则：

- 不再把“前端按模块上传”作为主流程。
- `M1/M2` 不从零动态生成，而是模板套用和字段替换。
- `M5` 根据项目资料找案例，不编造案例。
- `M6` 固定模板。
- `M3/M4` 后续再做动态生成。

## 3. 新版 Demo 工作流

1. 用户创建项目。
2. 用户填写基础信息。
3. 用户统一上传全部项目资料。
4. 后端保存文件。
5. 后端解析文件文本和元数据。
6. 后端判断资料角色和可能服务模块。
7. 后端识别项目类型。
8. 后端选择 `M1/M2` 固化模板。
9. 后端提取标签并匹配 `M5` 案例。
10. 后端选择 `M6` 固定模板。
11. 前端展示识别结果。
12. 用户确认或修正项目类型、模板和案例。
13. 后端套模板、替换字段、填充案例。
14. 后端按 `M1/M2 -> M5 -> M6` 合并最终 PPTX。
15. 用户下载最终 PPTX。

推荐总任务状态：

```text
待上传 -> 资料解析中 -> 类型识别中 -> 案例匹配中 -> 待确认 -> 生成中 -> 合并中 -> 完成/失败
```

推荐文件解析状态：

```text
pending -> parsed -> classified -> failed
```

## 4. API 契约

新版主流程接口：

| 方法 | 路径 | 用途 |
|---|---|---|
| `GET` | `/api/projects` | 项目列表 |
| `POST` | `/api/projects` | 创建项目 |
| `GET` | `/api/projects/{project_id}` | 项目详情 |
| `POST` | `/api/projects/{project_id}/files` | 统一上传全部项目资料，支持多文件 |
| `POST` | `/api/projects/{project_id}/analyze` | 解析资料、归类资料、识别项目类型、选择模板、匹配案例 |
| `GET` | `/api/projects/{project_id}/classification` | 查询系统识别结果 |
| `POST` | `/api/projects/{project_id}/classification/review` | 人工确认或修正识别结果 |
| `POST` | `/api/projects/{project_id}/generate` | 按确认结果生成 PPTX |
| `GET` | `/api/projects/{project_id}/task` | 查询任务状态 |
| `GET` | `/api/projects/{project_id}/download` | 下载最终 PPTX |
| `GET` | `/api/cases` | 查询案例库 |

兼容旧 Demo 的接口：

| 方法 | 路径 | 说明 |
|---|---|---|
| `POST` | `/api/projects/{project_id}/modules/{module_id}/files` | 可暂时保留，但新版前端不调用 |
| `GET` | `/api/assets?module_id=M1` | 可作为旧资产调试接口，不作为新版主流程必需接口 |

## 7. 验收标准

- 前端没有“分模块上传材料”主流程。
- 前端不要求用户选择资料属于 M 几。
- 前端支持统一多文件上传。
- 后端统一上传接口可保存多个文件。
- 文件记录不依赖前端传入 `module_id`。
- 系统能输出资料分类结果。
- 系统能识别项目类型。
- 系统能为 `M1/M2` 选择固化模板。
- 系统能为 `M5` 推荐案例。
- 系统能为 `M6` 选择固定模板。
- 用户可以确认或修正识别结果。
- 最终 PPTX 合并顺序为 `M1/M2 -> M5 -> M6`。
- `M3/M4` 不参与本阶段生成。
- LLM、Embedding、pgvector 不可用时，Demo 仍可通过 Mock 或关键词 fallback 跑通。
